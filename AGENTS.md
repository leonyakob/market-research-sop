# 品牌策划全案 SOP 项目规则

> 本项目特定规则，优先级高于全局 AGENTS.md。
> 通用规则请参考：~/.config/opencode/AGENTS.md

---

## 一、项目定义

本项目为品牌策划全案 SOP 构建项目，目标是将品牌策划流程沉淀为可迭代、可复用、可版本管理的结构化方法论体系。

项目采用：模块化结构 + Git/GitHub 历史保留 + 单主文件工作区模式。

---

## 二、结构规则（Structure Rules）

1. 采用扁平化交付结构，所有 SOP 文件统一存放在 `交付文档/` 目录，不创建模块子目录。

2. SOP 文件命名必须遵循统一规范：

   ```
   {模块号}-{模块名称}SOP.md
   ```

   示例：
   - `1.1-行业研究SOP.md`
   - `1.3-消费者研究SOP.md`

3. 版本演进信息必须内嵌在 SOP 文件附录中管理，不创建独立 README 文件，不创建 v1/v2/v3 并行文件。

---

## 三、版本管理规则（Versioning Rules）

1. 工作区策略：同一模块在 OpenCode 工作区只保留一个当前主文件，不保留版本后缀文件。

2. 历史保留：历史版本通过 Git / GitHub 提交记录和文档附录中的“版本历史”保留。

3. 每次对 `交付文档/` 下 SOP 主文件做出有效内容修改后，必须执行 Git commit 并 push 到 GitHub。

4. 有效内容修改定义：对 SOP 文件内容产生实质性变更的操作，必须满足以下任一类：
   - 新增、删除、修改章节内容
   - 调整分析框架或方法论
   - 更新数据获取路径、来源、参数或引用方式
   - 修改输出格式、判断标准或质量检查标准

5. 下列操作单独发生时，不视为有效内容修改，不触发 commit + push 要求：
   - 格式调整，例如缩进、空行、标点
   - 错别字修正
   - 注释补充
   - 仅新增版本历史章节记录

6. 如“版本历史”更新属于恢复流程或优化流程的一部分，且同次操作中同时包含有效内容修改，则必须随该次有效内容修改一并 commit + push。

7. 恢复规则：使用 Git / GitHub 历史恢复旧版本到当前主文件后，必须在该文件“版本历史”章节新增一条记录，并执行 Git commit 与 push。

8. 版本历史记录必须写明：版本标识、恢复或优化原因、关键变更点。

---

## 四、版本历史内嵌规范

每个 SOP 文件必须在附录（第八章）包含“版本历史”章节，用于记录版本演进逻辑。

### 版本历史标准结构：

```markdown
# 行业研究 SOP 版本说明

## v1（初版）
初版流程，覆盖基础行业分析逻辑

## v2（优化版）
优化点：
- 增加竞品分析维度
- 调整信息收集路径
- 明确输出格式

## v3（结构重构版）
优化点：
- 重构分析框架
- 提升模块复用性
```

## 五、行为规则（AI 执行逻辑）

### 0. 通用确认规则

- 所有涉及文件修改的操作，必须先说明修改范围或输出变更计划。
- AI 输出问题、方案或计划后，必须立即停止生成，等待用户明确确认。
- 未获用户明确确认前，不得直接修改文件，不得使用默认值，不得自行跳过确认步骤。
- 仅当用户明确回复“确认”“使用默认值”或“跳过”等可执行指令时，方可继续后续修改流程。

### 1. 当用户说：“优化当前 SOP”

系统必须执行：

- 澄清优化目标
- 澄清优化范围
- 输出优化方案或修改计划
- 等待用户确认
- 仅在用户确认后修改当前主文件
- 修改后更新 SOP 文件内的版本历史章节，新增该版本记录并总结优化点
- 执行 Git commit 与 push

### 2. 当用户说：“回到某个版本（如 v1 / 行业研究初版）”

系统必须执行：

- 使用 Git / GitHub 历史定位目标版本
- 使用 Git / GitHub 历史恢复旧版本到当前主文件
- 在当前主文件“版本历史”章节新增回溯记录，并标注“回溯使用版本”
- 执行 Git commit 与 push

### 3. 当用户说：“完成一个阶段 / 保存当前版本”

系统必须执行：

- 检查当前主文件是否存在有效内容修改
- 如存在有效内容修改，补写版本历史章节
- 执行 Git commit 与 push
- 如不存在有效内容修改，仅允许补充版本历史记录，不得虚构内容变更，且该补充单独发生时不触发 commit + push

### 4. 当用户请求修改规则文本本身时

系统必须执行：

- 先说明拟修改范围
- 先输出变更计划
- 等待用户确认后再编辑
- 未确认前不得直接改写规则文件

## 六、GitHub 同步规则

所有版本操作必须同步到 GitHub 仓库：

1. 每次有效内容修改 → commit + push。

2. commit message 必须包含：
   - 模块名
   - 变更摘要

3. 版本号可选，仅在需要强调版本演进时添加。

4. 示例：

   ```
   feat: 行业研究SOP（竞品分析增强 + 流程优化）
   ```

5. push 失败时，必须保留本地 commit，并提示用户检查网络、权限或分支冲突；详细处置以第十二章异常边界处理为准。

---

## 七、核心设计原则

1. 版本不是覆盖，而是资产积累。
2. 每一个 SOP 都是可复用模块。
3. 版本历史是方法论演进记录，内嵌于 SOP 文件附录。
4. GitHub 是知识资产库，不是代码仓库。
5. 数据驱动，所有分析结论必须有数据支撑，数据来源必须可追溯。

---

## 八、SOP v3.0 执行规则

### 8.1 数据获取
- 所有 SOP 模块的数据获取必须通过 `.sop/config/data-acquisition.yaml` 配置。
- 当前使用 Exa/Tavily 外部搜索，未来切换为自建数据源。
- 切换方式：修改配置文件中的 `primary` 字段，SOP 文件无需改动。
- 数据调用使用自然语言查询模板，不使用固定搜索词。

### 8.2 数据融合
- 外部数据与内部数据是“融合验证”关系，不是“覆盖替代”关系。
- 详见 `交付文档/SOP-融合与冲突处理规则.md`。

### 8.3 内部文件处理
- 内部文件标注只写文件名，不写路径。
- 详见 `.sop/config/internal-files.yaml`。

### 8.4 模板使用
- 所有 SOP 模块必须基于 `交付文档/SOP-通用模板.md` 的骨架结构创建。
- 模板仅定义章节骨架，具体内容由各模块自定义。

### 8.5 强制交互规则（关键）
- 所有 SOP 模块中的参数确认步骤，遵循第五章通用确认规则执行。

## 九、SOP 章节前置声明规范

1. 每个章节开头必须包含以下两行前置声明：

   ```markdown
   > **本章包含**：...
   > **所需数据/资料**：...
   ```

2. “数据”与“资料”必须区分使用：
   - 数据，用于可量化、可验证的信息
   - 资料，用于文档、案例、规范、资产材料

3. 前置声明应简洁明了，可根据章节复杂度灵活控制篇幅，但不得缺失。

## 十、思考逻辑梳理规范

1. 触发条件：每次 SOP 模块交付完成后，必须同步产出思考逻辑梳理文档。

2. 存放路径：

   ```
   思考逻辑梳理/{模块号}-{模块英文名}-thinking-logic.md
   ```

3. 文档结构必须包含以下五部分：

   - 设计原则演进
   - 关键决策点
   - 自我纠错记录
   - 版本演进总结

## 十二、异常边界处理

1. Push 失败处理：
   - 必须保留本地 commit
   - 提示用户检查网络连接、GitHub 权限或分支冲突
   - 由用户确认后，方可重试 push

2. 工作区无关改动处理：
   - 检测到 `交付文档/` 外的文件改动时，先暂停 SOP 相关操作
   - 提示用户先 stash 或提交无关改动
   - 用户确认后再继续
   - 恢复 stash 仅在用户确认且操作安全时执行

3. 双输入缺失处理：
   - 检测到双输入模块缺少数据或资料任一来源时，必须暂停
   - 明确提示用户补充缺失来源
   - 不得使用默认值或假设值

4. 用户拒绝确认处理：
   - 用户拒绝优化方案或修改计划时，不执行任何修改
   - 保留当前状态
   - 结束本次修改流程

5. 版本历史冲突处理：
   - 恢复旧版本时发现版本历史记录不完整，必须暂停
   - 先补充缺失的版本记录，再继续
   - 基于现有信息推断版本语义时，必须标注“推断”
