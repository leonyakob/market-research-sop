## 二、核心设计原则演进

### 原则 1：求真务实高于一切

**触发事件**：初始测试发现 AI 编造数据、来源模糊。

**认知升级**：
```
初始状态：AI 生成看似专业的报告，但数据无来源
核心认知：没有数据来源的 SOP 只是"文字游戏"，必须建立信任基石
```

**设计原则**：
- 所有事实性信息必须来源于可验证资料
- 每一条关键结论附带来源说明
- 禁止无依据推测，信息不足时必须声明

### 原则 2：融合验证 > 简单覆盖

**触发事件**：用户质疑"内部数据直接覆盖外部数据"的逻辑。

**认知升级**：
```
错误逻辑：新数据覆盖旧数据
正确逻辑：数据冲突蕴含业务洞察（如统计口径差异），应并行比对、预警决策
```

**设计原则**：
- 外部数据与内部数据是"融合验证"关系
- 差异 < 5% → 增强置信度
- 差异 > 10% → 触发预警，请求用户决策

### 原则 3：配置与逻辑分离

**触发事件**：技术人员指出 SOP 写死搜索词，无法对接未来内部数据库。

**认知升级**：
```
硬编码：SOP 里写"去欧睿搜 XX 词"
解耦：SOP 定义"我要什么数据"，配置文件定义"去哪找、怎么找"
```

**设计原则**：
- SOP 中只保留自然语言查询模板
- 数据源配置独立于 `data-acquisition.yaml`
- 实现热切换能力（Exa ↔ Internal API）

### 原则 4：渐进式构建 > 一次性填空

**触发事件**：用户拒绝"启动时一次性问 8 个问题"的方案。

**认知升级**：
```
工程师思维：假设用户启动前知道所有答案
顾问思维：用户往往处于探索期，需要 AI 通过调研提供选项
```

**设计原则**：
- Phase 0 极简启动（只问品类）
- Phase 1 调研后生成带数据的选项
- Checkpoint 1 用户基于数据做选择

### 原则 5：强制交互（Hard Stop）

**触发事件**：AI 经常无视"等待 10 分钟"指令，自作主张使用默认值。

**认知升级**：
```
时间指令："10 分钟后..." → AI 无法感知时间，倾向于直接完成任务
行为指令："必须立即停止生成" → AI 知道"停止"才是正确行为
```

## 三、关键决策点与理由

| 决策点 | 选项 | 选择 | 核心理由 |
|--------|------|------|---------|
| 参数确认机制 | 自动决定/等待确认 | 强制交互 + Hard Stop | 防止 AI 自作主张，确保用户控制权 |
| 数据获取方式 | 固定搜索词/自然语言调用 | 自然语言调用 + 配置文件 | 支持热切换，适配未来内部数据库 |
| 数据冲突处理 | 覆盖/融合验证 | 融合验证（非覆盖） | 保留数据差异背后的业务洞察 |
| 通用模板内容 | 包含行业示例/纯占位符 | 纯占位符骨架 | 防止后续模块产生幻觉 |
| 启动交互模式 | 一次性问 8 个/渐进式构建 | 渐进式构建 | 符合用户探索期心理，降低门槛 |
| 检查清单机制 | 全部通过/分级通过 | P0/P1/P2 分级 | 提高容错性，避免流程卡死 |
| 接口定义时机 | 先定义/后定义 | 内部设计完成后定义 | 确保 I/O 与模块实际产出匹配 |

## 四、自我纠错记录

| 错误 | 发现方式 | 修正 |
|------|---------|------|
| 通用模板包含 PEST/五力 | 用户质疑后续模块会幻觉 | 清除所有行业专属内容，改为占位符 |
| 启动时一次性问 8 个问题 | 用户指出体验差 | 改为渐进式上下文构建 |
| "10 分钟未回复用默认值"失效 | 实验发现 AI 直接跳过 | 改为"输出问题后必须立即停止生成" |
| 内部数据直接覆盖外部数据 | 用户质疑不合理 | 改为融合验证逻辑 |
| 依赖声明未明确"无"是合法答案 | 用户质疑 | 优化占位符为 `[前置模块/无]` |
| 未定义模块接口 | 用户指出无法自动化编排 | 增加"九、接口定义"章节 |
