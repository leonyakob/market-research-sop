# 1.4 趋势分析 SOP 构建思考逻辑

> **Session**: 1.4 趋势分析 SOP 封版交付沉淀
> **时间**: 2026-04-15
> **目标**: 复盘 1.4 趋势分析 SOP v3.0 从中间态设计到最终封版的收敛路径，沉淀字段边界、判断规则与跨模型稳定性策略。

## 一、决策演进树

```text
起点：1.1（行业）/ 1.2（竞争）/ 1.3（消费者）已完成或已形成接口，需要设计 1.4，把“变化信号”转成可供 1.5 与 2.x 消费的趋势模块
  ↓
核心问题 1：1.4 的模块定位是什么？
  → 备选：趋势综述 / 数据补充章节 / 下游判断接口
  → 决策：1.4 不是“罗列趋势”，而是把 1.1/1.2/1.3 的变化信号转译为可行动的趋势判断
  ↓
核心问题 2：1.4 属于什么模块类型？
  → 查证：AGENTS.md 明确将 1.4 列为“数据优先”；Prompt 模板将 1.4 列为“数据研究型”
  → 决策：1.4 以可验证数据为主，不走资料优先或纯描述路径
  ↓
核心问题 3：1.4 的输入来自哪里？
  → 查证：1.3 已显式输出 segment_tensions_to_track / emerging_behaviors / cultural_signals_by_segment 给 1.4
  → 查证：1.2 条件输出 C4（竞品动态监测点）同时供 1.4 / 1.5 使用
  → 决策：1.4 必须承接 1.3 的消费者变化信号，同时吸收 1.2 的竞争动态，并以 1.1 的行业背景作为解释层
  ↓
核心问题 4：1.4 的主结构如何组织？
  → 模板给出的 A/B/C 选项都不完全适合 SOP 流程化表达
  → 决策：采用混合流程型结构
     1) 趋势信号来源与获取
     2) 趋势识别与聚类
     3) 伪趋势过滤 / 有效性判断
     4) 时间分层
     5) 趋势解释与影响分析
     6) 趋势输出与下游接口
  ↓
核心问题 5：趋势识别范围怎么定？
  → 备选：只做消费 / 消费+行业 / 四类并重 / 四类全收后分优先级输出
  → 用户确认：先收全四类，但输出时按“消费 / 行业优先”排序
  → 决策：采集层全收（宏观/行业/消费/渠道传播），输出层主次分明（消费 > 行业 > 渠道传播 > 宏观）
  ↓
核心问题 6：是否必须区分短期 / 中期 / 长期？
  → 用户追问：如果定义模糊，会导致 AI 自由发挥、跨模型不稳定
  → 决策升级：保留时间分层，但不再让 AI 用自然语言直觉判断；长期类改写为“长期观察方向”，不与短/中期并列为同等级正式趋势
  ↓
核心问题 7：长期类如何避免“预测未来”式幻觉？
  → 反思：不能让 AI 判断“尚未明显出现但未来重要”
  → 决策：长期类采用“排除短期/中期 + 满足观察池准入条件”的方式，只进入观察池/预警清单，不进入正式趋势主表
  ↓
核心问题 8：时间分层能否独立成立？
  → 用户指出：若不先过滤伪趋势，时间分层会把噪音也分进去
  → 决策：伪趋势过滤 / 有效性判断必须前置，且与时间分层成套设计
  ↓
核心问题 9：规则描述中哪些词会导致模型漂移？
  → 用户逐条质疑：较强延续性 / 有价值 / 结构性 / 相近 / 同类 / 同方向 / 相关 / 初步 / 不稳 等词均缺乏边界
  → 决策：全面转向字段化、标签化、门槛化、缺口化表达
  ↓
核心问题 10：候选对象如何避免聚类与判断时出现主观解释？
  → 决策：引入最小结构字段（target_object / direction_label / driver_label / downstream_mapping / source_categories 等）
  → 决策：聚类基于字段一致，不基于语义“相近”
  → 决策：来源独立性基于同一 target_object + direction_label 组合，不再使用“同方向”自然语言判定
  ↓
核心问题 11：内部资料为什么不能单独支撑正式归类？
  → 分析：内部资料天然带有样本偏差、场景偏差、项目特异性
  → 决策：内部资料可触发候选对象、补充解释、做交叉验证，但短期/中期/长期观察方向都至少需要 1 类外部来源支持
  ↓
核心问题 12：1.4 输出该如何服务下游？
  → 外部方法论与内部接口共同指向：1.4 不能只写趋势综述，必须形成可交接资产
  → 决策：输出层至少区分“正式趋势主表”“长期观察池”“淘汰/暂存列表”；正式趋势再根据 downstream_mapping 交给 1.5 或 2.x
  ↓
当前封版结论：1.4 的核心不在“写一份趋势报告”，而在建立一套把变化信号稳定转为趋势对象、再转为下游接口的确定性流程；封版阶段进一步把模糊词、悬空字段与多义映射全部收口为可执行规则。
```

## 二、设计原则演进

### 原则 1：确定性优先于“趋势洞察感”

**触发事件**：当开始定义短期 / 中期 / 长期趋势时，出现了“较强延续性”“结构性变化”“对中期布局有价值”等表述。用户立即指出：这些词会把判断偷偷交给 AI，不同模型会产生不同结果。

**原则落地**：
- 时间分层不能建立在形容词上
- 所有判断尽量改成字段、标签、门槛、缺口项
- AI 只能执行规则，不能靠感觉归类趋势

### 原则 2：先过滤噪音，再做趋势判断

**触发事件**：在先讨论时间分层时，用户指出若不先做伪趋势过滤，短中长期分类会把热点、单次事件、单一个案一起吸纳进来。

**原则落地**：
- 有效性判断必须早于时间分层
- 未通过有效性判断的对象，不得进入短期趋势、中期趋势或长期观察方向
- 长期观察池不是噪音回收站

### 原则 3：长期类不做“未来结论”，只做“观察准入”

**触发事件**：用户追问“尚未明显出现”“仅早期信号”如何判断，指出 AI 不应承担预测未来的职责。

**原则落地**：
- 将“长期趋势”重命名为“长期观察方向”
- 长期类只在不满足短期/中期且满足观察池准入条件时成立
- 长期类只进入观察池/战略预警列表，不进入正式趋势主表

### 原则 4：聚类与归类都必须依赖结构化字段，不依赖语义相似

**触发事件**：用户质疑“相近驱动逻辑”“同类影响”“同方向变化”“相关动作/相关表现/相关变化”等词汇是否仍会导致模糊与漂移。

**原则落地**：
- 候选对象先字段化，再聚类
- 聚类仅基于 target_object / direction_label / driver_label / downstream_mapping 等固定字段
- 来源独立性不再写“同方向”，而是要求相同的 target_object + direction_label 组合
- “相关”类词汇全部删除，改成固定动作标签、表现标签或方向标签

### 原则 5：正式趋势必须可复核、可迁移，内部资料不能单独定案

**触发事件**：用户追问内部资料为何不能单独支撑正式归类，要求规则符合 SOP 的科学性与跨项目可复用性。

**原则落地**：
- 内部资料定位为信号触发、解释补充、交叉验证
- 正式归类至少需要 1 类外部来源支持
- 这样可以降低内部样本偏差被误判为市场趋势的风险

### 原则 6：1.4 的价值不在“多写趋势”，而在“把趋势交给下游”

**触发事件**：在内部检索中，1.3 已显式输出给 1.4；同时 Prompt 模板明确 1.4 的输出必须服务 1.5 与品牌战略模块。

**原则落地**：
- 1.4 不是终点报告，而是市场研究阶段的判断中枢
- 输出必须带 downstream_mapping，决定进入 1.5 还是 2.x
- 趋势对象必须成为“下游可消费接口”，而不是只停留在研究描述层

## 三、关键决策点

| 决策点 | 备选 | 最终选择 | 核心理由 |
|---|---|---|---|
| 1.4 模块定位 | 趋势综述 / 研究补充 / 下游判断接口 | 下游判断接口 | 要把变化信号转译成可行动趋势判断，而非再写一份综述 |
| 模块类型 | 资料优先 / 数据优先 / 双输入 | 数据优先 + 数据研究型 | AGENTS 与 Prompt 模板已有明确分类 |
| 上游关系 | 只接 1.3 / 只接外部数据 / 多上游融合 | 1.1+1.2+1.3 融合，主接 1.3 与 1.2 | 1.3 提供消费者变化信号，1.2 提供竞争动态，1.1 提供行业背景 |
| 主结构 | A/B/C 模板单选 | 混合流程型结构 | 更适合把“信号→判断→交接”写成 SOP 流程 |
| 趋势识别范围 | 消费优先 / 消费+行业 / 四类并重 / 四类全收后排序 | 四类全收，输出按消费/行业优先排序 | 保完整性，同时避免失焦 |
| 时间体系 | 短/中/长期趋势并列 / 不分时间 / 保留长期但改写 | 短期趋势 + 中期趋势 + 长期观察方向 | 避免把长期项误写成确定性结论 |
| 时间分层依据 | 形容词描述 / 规则判断 | 规则判断 | 降低 AI 主观解释空间 |
| 有效性判断位置 | 可选 / 后置 / 前置强制 | 前置强制 | 先过滤热点/个案/噪音，才能做时间分层 |
| 候选对象表达方式 | 自然语言归纳 / 字段化对象 | 字段化对象 | 便于聚类、复核、跨模型稳定 |
| 聚类标准 | 相近 / 同类 / 字段一致 | 字段一致 | 语义相似会导致模型漂移 |
| 来源独立性定义 | 多来源同方向 | 相同 target_object + direction_label 组合跨来源类别出现 | 避免“同方向”自然语言判断 |
| 内部资料角色 | 可单独定案 / 辅助来源 | 辅助来源 | 内部样本不能单独代表市场趋势 |
| 下游输出层 | 单一趋势表 / 多层输出 | 正式趋势主表 + 长期观察池 + 淘汰/暂存列表 | 便于 1.5 / 2.x 分层消费 |

## 四、自我纠错记录

| 问题/风险 | 发现方式 | 修正动作 |
|---|---|---|
| 将 1.4 初步理解为“趋势综述模块” | 对照 1.3 / 1.2 / Prompt 模板后发现定位偏浅 | 改为“变化信号 → 趋势判断 → 下游接口”的判断中枢 |
| 先讨论时间分层，未同步给出伪趋势过滤规则 | 用户明确指出两者必须配套使用 | 改为“有效性判断前置、时间分层后置”的成套规则 |
| 使用“较强延续性”“结构性变化”“有价值”等模糊词 | 用户逐条追问定义边界与执行一致性 | 逐步改写为字段、标签、门槛、缺口项 |
| 将“长期趋势”与短中期并列 | 用户质疑 AI 如何判断尚未明显出现的趋势 | 改名为“长期观察方向”，只入观察池 |
| 聚类规则里保留“相近驱动逻辑”“同类影响” | 用户指出 AI 仍会主观解释 | 改为 target_object / direction_label / driver_label / downstream_mapping 的固定字段比对 |
| 来源独立性中使用“同方向变化” | 用户质疑“同方向”仍是模糊词 | 改为相同 target_object + direction_label 组合 |
| 多主体 / 多场景支持里使用“相关动作/表现/变化” | 用户指出“相关”是典型偷渡词 | 改成固定动作标签 / 表现标签 / 方向标签 |
| 待确认条件中出现“初步可见但证据不稳” | 用户指出“初步”“不稳”仍无法执行 | 改为明确缺口：来源独立性不足 / 重复出现性不足 / 下游映射仅暂不明确 |
| 规则输出被拆成多轮补丁式给出 | 用户明确要求成套规则一次性输出 | 后续改为总流程版整合：候选对象 → 有效性判断 → 时间分层 → 输出接口 |

## 五、版本演进总结

| 版本 | 核心能力 | 价值 |
|---|---|---|
| v0.1（中间态） | 明确 1.4 的模块定位、主结构、四类采集范围、时间分层框架、长期观察池、有效性判断前置、候选对象字段化、来源独立性与聚类规则去模糊方向 | 将“趋势分析”从模糊报告型研究，推进为“可过滤、可分层、可交接”的规则化判断模块 |
| **v3.0（封版）** | 完成字段边界收口：固定 `trend_name` 规则与中文方向映射、锁定 `source_categories` 枚举、模板化 `required_next_evidence`、固定 `review_window`、补齐 `driver_label`/`validity_status`/`evidence_summary`、写死计数字段与 `external_support` 判定口径，并将长期项、观察池与下游交接边界全部显式化 | 将 1.4 从“方法方向已确定”推进为“AI 可执行、可复核、跨模型更稳定”的正式交付模块 |

**本次沉淀结论**：1.4 的关键不在“多收集趋势材料”，而在把来自 1.1 / 1.2 / 1.3 的变化信号收敛为字段化候选对象，经有效性判断与时间分层后，输出成 1.5 与 2.x 可直接消费的趋势接口。封版阶段的核心工作不是继续扩展结构，而是系统性收口所有会导致执行漂移的模糊词、悬空字段与边界缺口，使该模块真正具备科学性、可复用性与稳定执行性。
