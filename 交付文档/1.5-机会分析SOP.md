## 前置声明

### 1.5特异性求真务实原则（5条规则）

AI在执行本SOP时，**必须严格遵循以下规则**：

1. **所有机会判断必须来源于可验证的上游模块输出或外部数据，不得凭语言感觉补全机会结论**。
2. **每一个正式机会项必须附带来源模块、证据ID、筛选判定原因与优先级判定依据**。
3. **禁止将未经筛选验证的上游信号、假设或观察直接写为正式机会**。
4. **当证据不足以支撑机会筛选通过时，必须标注为incomplete或not_included，不得补全或猜测**。
5. **如不同来源存在冲突，必须按`~/github/market-research-sop/交付文档/SOP-融合与冲突处理规则.md`标注差异并处理，不得擅自平滑为单一结论**。

## 一、触发条件

### 1.1 何时执行本模块

本模块在以下场景触发执行：

- 已完成 1.1 / 1.2 / 1.3 / 1.4，需将上游信号整合为可筛选、可排序、可交接的机会判断
- 在进入 2.x 品牌战略模块前，需要明确哪些机会进入正式优先级清单，哪些进入观察池
- 在制定品牌定位、目标市场选择、差异化策略前，需要形成可复用、可下游消费的机会接口

### 1.2 前置模块完成状态

| 前置模块 | 完成状态要求 | 验证方式 |

| 1.1 行业研究 | 必须完成 | 检查 `.sop/context/1.1-output.json` 或对应正式输出是否存在 |
| 1.2 竞争分析 | 必须完成 | 检查 `.sop/context/1.2-output.json` 或对应正式输出是否存在 |
| 1.3 消费者研究 | 必须完成 | 检查 `.sop/context/1.3-output.json` 或对应正式输出是否存在 |

- 大小限制：50MB

### 2.2 AI需确认（3-5个，含默认值机制）

| 参数 | AI询问方式 | 选项示例 | 默认值（仅用户明确要求时使用） |
|------|------------|----------|------------------------------|
| 机会输出粒度 | "本轮需要输出标准版还是扩展版机会卡？" | 标准版 / 扩展版 | 标准版 |
| 下游优先级 | "本轮机会输出优先服务 2.1 品牌定位，还是同时服务 2.x 全模块？" | 仅2.1 / 2.1+2.2+2.3 | 2.1+2.2+2.3 |
| 观察池参与程度 | "1.4观察池中的对象是否允许参与机会候选触发？" | 不参与 / 参与触发 | 参与触发 |

**执行规则（强制交互）**：

1. **必须暂停**：AI 输出参数确认问题后，**必须立即停止生成**，不得继续执行搜索、分析或字段填充。
2. **等待回复**：等待用户明确回复（选择选项、输入自定义值或回复"使用默认值"）。
3. **禁止自作主张**：严禁在未获用户确认前使用默认值、补全参数或替用户推断任务口径。
4. **默认值触发**：仅当用户明确回复"使用默认值"或"跳过"时，方可使用默认值。
5. **报告中标注确认情况**：如"本研究参数已确认：下游优先级=2.1+2.2+2.3；观察池参与=参与触发"。

### 3.1 外部数据源调用

**配置引用**：`.sop/config/data-acquisition.yaml`

**执行逻辑**：

```
读取 1.1 / 1.2 / 1.3 / 1.4 上游输出 → 生成自然语言查询 → 调用主数据源
  → 若主数据源失败 → 调用 fallback
  → 返回数据 → 提取机会信号 → 字段化标注 → 进入机会池生成
```

**数据调用指令模板**（AI按此格式生成查询）：

| 数据维度 | 自然语言查询模板 | 预期返回内容 |
|---------|----------------|-------------|
| **市场机会信号** | `获取{品类}在{地理范围}的市场空白点、未满足需求、新兴细分市场机会，包含机会描述、目标人群、市场规模估算` | 市场机会列表 + 目标人群 + 规模估算 |
| **竞争机会信号** | `获取{品类}在{地理范围}的竞争空白区域、竞品薄弱环节、差异化切入点，包含机会类型、切入条件、竞争态势` | 竞争机会列表 + 切入条件 + 竞争评估 |
| **人群机会信号** | `获取{品类}目标人群中的高增长群体、高痛点低满足群体、新兴行为群体，包含人群特征、需求强度、增长潜力` | 人群机会列表 + 特征描述 + 潜力评估 |
| **趋势机会信号** | `获取{品类}相关趋势中的可转化机会、趋势驱动的市场变化、趋势孕育的新需求，包含趋势关联、机会窗口、转化路径` | 趋势机会列表 + 窗口期 + 转化路径 |

**占位符替换规则**：
- `{品类}` → 用户输入的目标品类
- `{地理范围}` → 确认的地理范围

**文件读取规则**：

| 文件大类 | 读取方式 | 标注规范 | 隐私级别 |
|----------|----------|----------|---------|

| 竞品情报类 | 来源标注→时效性检查→交叉验证 | 【内部-竞品情报-文件名】 | 中/高 |

### 3.4 AI自主补充查询

**补充触发条件表**：

| 条件 | 判断方法 | 阈值 | 补充查询指令 |
|------|----------|------|-------------|
| 上游字段缺口 | 1.1 / 1.2 / 1.3 / 1.4 任一关键字段缺失 | ≥1个关键字段缺失 | 补充查询：缺失字段对应维度 + 当前品类 |
| 机会证据不足 | 候选机会仅有1类来源支持 | 仅1类来源 | 补充查询：第二来源类别验证 |
| 规模数据缺失 | 机会项缺少市场规模估算 | 无规模数据 | 补充查询：机会对象 + 市场规模 + 增速 |
| 竞争态势不明 | 机会项缺少竞争封锁评估 | 无竞争评估 | 补充查询：机会对象 + 竞争格局 + 进入壁垒 |

**数据调用失败处理流程**：

```
主数据源调用失败 → 自动降级fallback（Tavily）
  → 同义词扩展查询（如"机会"→"空白点"→"切入点"）
  → 扩大查询范围（如具体场景→所属消费场景大类）
  → 若仍无结果 → 标注"信息不足"并说明已尝试路径
  → 继续执行其他对象，不阻塞流程
```

## 四、输出格式

### 4.1 报告结构（6个部分）

**一、机会信号来源与获取**
- 1.1 上游输入概览：1.1 / 1.2 / 1.3 / 1.4 已读取字段、可用性状态、缺口说明
- 1.2 外部补充数据概览：补充查询维度、来源类别、时效性说明
- 1.3 机会池生成说明：机会项数量、来源分布、类型分布

**二、机会类型分类**
- 2.1 机会类型分布：6类机会的数量、占比、来源模块
- 2.2 类型分类说明：分类依据、边界案例、未分类原因

**三、机会筛选与评估**
- 3.1 筛选结果总览：通过 / 观察池 / 不纳入的数量分布
- 3.2 四维评分结果：需求真实度、增长潜力、竞争可切入性、战略承接性评分分布
- 3.3 一票否决情况：证据链不完整、重复机会、边界冲突

**四、优先级排序**
- 4.1 P1级机会（立即推进）：总分≥150且所有维度≥1的机会清单
- 4.2 P2级机会（计划推进）：120≤总分<150的机会清单
- 4.3 P3级机会（观察储备）：80≤总分<120的机会清单
- 4.4 not_now级（暂不推进）：总分<80或任一维度=0的机会清单

**五、观察池与风险提示**
- 5.1 观察池清单：进入观察池的机会项、观察原因、复查周期
- 5.2 风险提示表：需求风险、竞争风险、执行风险、战略风险
- 5.3 关联假设表：需求假设、竞争假设、能力假设、资源假设

**六、机会输出与下游接口**
- 6.1 正式机会主表：标准化机会卡输出
- 6.2 观察池清单：观察对象与监测条件
- 6.3 淘汰/暂存列表：不进入正式输出的机会与原因
- 6.4 下游交接：2.1 / 2.2 / 2.3 可直接消费字段

### 4.3 数据来源标注规范

**数据融合标注**（v3.0沿用）：

| 标注格式 | 含义 | 示例 |
|---------|------|------|
| `【外部-数据源名】` | 仅单一外部源 | `市场空白点识别【外部-艾瑞】` |
| `【外部-数据源名 + 内部-文件名】（已验证）` | 多源一致，差异<5% | ` underserved_segments确认【外部-益普索 + 内部-用户研究】（已验证）` |
| `【外部-数据源名 vs 内部-文件名】（差异X%，见冲突分析）` | 存在冲突，差异>10% | `目标人群规模估算存在差异【外部-欧睿 vs 内部-销售数据】（差异15%，见冲突分析）` |
| `【信息不足】` | 搜索、读取或交叉验证后仍无法形成有效证据 | `某细分机会仅单一来源记录【信息不足】` |

**字段填写规则**：
- `机会名称（opportunity_name）`：仅填写名词性机会名称，不得写结论句、比较句或带方向的判断句
- `机会类型（opportunity_type）`：仅允许使用6类固定枚举值
- `目标人群（target_segment）`：仅填写目标人群标识，不得混入策略建议
- `核心场景（core_scenario）`：仅填写核心场景标识
- `动作方向（action_direction）`：仅填写动作方向描述，不得写价值判断
- `下游映射（downstream_mapping）`：仅填写固定模块标识，如 `2.1 / 2.2 / 2.3`；不得写解释句

## 五、分析规则与决策标准

### 5.1 数据融合规则

**引用**：`~/github/market-research-sop/交付文档/SOP-融合与冲突处理规则.md`

**融合流程**：

```
并行获取多源数据 → 智能比对 → 冲突检测 → 融合/冲突处理 → 输出结果
```

**冲突处理优先级**：

| 冲突类型 | 处理策略 | 优先级 |
|----------|----------|--------|
| 高可信 vs 中可信 | 采用高可信 | P0 |
| 多源高可信冲突 | 标注差异，说明采用依据 | P0 |
| 时效性冲突 | 采用最新数据 | P1 |
| 统计口径冲突 | 标注口径差异 | P1 |

### 5.2 机会池生成规则

**上游输入→机会项标准化流程**：

```
读取 1.1 / 1.2 / 1.3 / 1.4 输出 → 提取机会信号 → 字段标准化 → 生成 opportunity_id → 进入机会池
```

**机会项字段最小集**：

| 字段 | 类型 | 填写规则 |
|------|------|----------|
| `机会编号（opportunity_id）` | string | 唯一编号，格式 `OP001` 起 |
| `机会名称（opportunity_name）` | string | 仅填写机会名称，不得包含判断、比较或结论性形容词 |
| `机会类型（opportunity_type）` | enum | `市场机会（market_opportunity） / 人群机会（segment_opportunity） / 产品机会（product_opportunity） / 渠道机会（channel_opportunity） / 传播机会（communication_opportunity） / 组合机会（combined_opportunity）` |
| `目标人群（target_segment）` | string | 目标人群标识 |
| `核心场景（core_scenario）` | string | 核心场景标识 |
| `问题缺口（gap_description）` | string | 问题缺口描述 |
| `动作方向（action_direction）` | string | 动作方向描述 |
| `来源模块（source_modules）` | string[] | 标注来自 1.1 / 1.2 / 1.3 / 1.4 的哪些模块 |
| `上游证据ID（upstream_evidence_ids）` | string[] | 对应上游模块中的证据标识 |

**来源模块标记规则**：
- `1.1`：来自行业研究的机会点与风险点
- `1.2`：来自竞争分析的竞争机会清单、定位图空白区域
- `1.3`：来自消费者研究的 underserved_segments、high_growth_segments、high_pain_low_solution_fit_segments、opportunity_hypotheses
- `1.4`：来自趋势分析的 trend_opportunity_areas、high_priority_trends、watchlist_items_with_opportunity_potential

**opportunity_id生成规则**：
- 格式：`OP` + 三位数字，从 `OP001` 开始递增
- 分配顺序：按机会项进入机会池的时间顺序分配
- 去重保留时：保留 opportunity_id 较小者

### 5.3 机会类型分类规则

**6类固定枚举**：

| 机会类型 | 英文标识 | 定义 | 适用场景 |
|----------|----------|------|----------|
| 市场机会 | market_opportunity | 基于市场结构变化、空白区域、未满足需求形成的机会 | 新细分市场、市场空白点、需求迁移 |
| 人群机会 | segment_opportunity | 基于特定人群的高增长、高痛点、低满足度形成的机会 | 新兴人群、 underserved人群、高价值人群 |
| 产品机会 | product_opportunity | 基于产品功能、形态、体验创新形成的机会 | 功能创新、形态创新、体验升级 |
| 渠道机会 | channel_opportunity | 基于渠道结构变化、新渠道涌现、渠道效率差异形成的机会 | 新渠道、渠道空白、渠道效率优势 |
| 传播机会 | communication_opportunity | 基于内容趋势、传播方式变化、媒介红利形成的机会 | 内容红利、新媒介、传播创新 |
| 组合机会 | combined_opportunity | 跨多个维度的复合机会，单一类型无法准确概括 | 市场+人群、产品+渠道、人群+传播等多维联合 |

**组合机会使用条件**：
- 仅用于多维联合成立、单一类型无法准确概括的机会
- 必须明确列出涉及的组合维度（如"市场+人群"、"产品+渠道"）
- 不得作为默认兜底类型使用
- 组合维度必须在2-3个之间，超过3个维度需拆分为多个机会项

### 5.4 机会筛选标准

#### 5.4.1 证据链完整性判定（一票否决项）

**判定逻辑**：

上游证据与问题缺口必须共享至少一个字段（`target_segment` 或 `core_scenario`），问题缺口与动作方向必须满足 `problem_object` 与 `action_object` 一致或 `action_object` 明确指向 `problem_object` 的解决。

**判定流程（6步）**：

1. 提取上游证据中的 `target_segment` 和 `core_scenario` 字段
2. 提取机会项中的 `target_segment` 和 `core_scenario` 字段
3. 检查是否至少共享一个字段（`target_segment` 相同 或 `core_scenario` 相同）
4. 提取问题缺口中的 `problem_object`（问题对象）
5. 提取动作方向中的 `action_object`（动作对象）
6. 检查 `problem_object` 与 `action_object` 是否一致，或 `action_object` 是否明确指向 `problem_object` 的解决

**判定结果**：
- 步骤3通过且步骤6通过 → 证据链完整
- 步骤3不通过 → 标记 `evidence_gap_mismatch`，输出状态 `not_included`，判定原因 `evidence_chain_incomplete`
- 步骤6不通过 → 标记 `gap_action_mismatch`，输出状态 `not_included`，判定原因 `evidence_chain_incomplete`

#### 5.4.2 重复机会判定（去重规则）

**主判定字段**：`target_segment` + `core_scenario` + `action_direction` 全部相同

**辅判定字段**：`gap_description` / 证据指向 / `opportunity_type` 至少一致一项

**判定逻辑**：
- 主判定字段全部相同 且 辅判定字段至少一致一项 → 判定为重复机会
- 仅主判定字段相同但辅判定字段均不一致 → 保留为独立机会，标注 `potential_overlap`

**保留规则**：
1. 证据条数更多者优先
2. 证据条数相同 → 证据类型数更多者优先
3. 证据类型数相同 → `opportunity_id` 较小者优先

**输出状态**：`not_included`
**判定原因**：`duplicate_opportunity`

#### 5.4.3 需求真实度评分（0-2分）

| 评分 | 判定标准 |
|------|----------|
| 0分 | 无证据支持需求存在 |
| 1分 | 有≥1条上游证据指向需求存在，但问题边界未完全闭合 |
| 2分 | 有≥2条来自不同上游模块的证据指向同一需求，且问题边界闭合 |

**问题边界闭合判定**：
- 需求存在性有证据支持
- 需求强度可量化或可比
- 需求持续性有时间维度证据

#### 5.4.4 增长潜力评分（0-2分）

| 评分 | 判定标准 |
|------|----------|
| 0分 | 无增长逻辑且无增长数据支撑 |
| 1分 | 有增长逻辑但仅有单一来源支撑，或增长路径依赖≥1个未验证前提 |
| 2分 | 有增长逻辑且有≥2条来源支撑，增长路径依赖前提≤1个且该前提可验证 |

**增长逻辑判定**：
- 市场规模增长趋势
- 人群规模增长趋势
- 消费频次/金额增长趋势
- 品类渗透率提升空间

#### 5.4.5 竞争可切入性评分（0-2分）

| 评分 | 竞争封锁标记（competition_block_flag） | 新增前置条件数（new_prerequisite_count） |
|------|----------------------------------------|----------------------------------------|
| 0分 | blocked | ≥3 |
| 1分 | partial_block | 1-2 |
| 2分 | open | 0 |

**竞争封锁标记定义**：
- `open`：无显著竞争封锁，进入壁垒可突破
- `partial_block`：存在部分竞争封锁，需特定条件方可突破
- `blocked`：竞争封锁严重，进入壁垒极高

**新增前置条件数**：指我方当前不具备、必须新增才能切入该机会的条件数量（如特定资质、渠道关系、供应链能力等）。

#### 5.4.6 战略承接性评分（0-2分）

| 评分 | 与品牌当前战略方向一致性 | 新增假设数（new_assumption_count） | 上游核心输出承接性 |
|------|--------------------------|------------------------------------|---------------------|
| 0分 | 冲突 | ≥2 | 无法承接 |
| 1分 | 一致 | 1 | 部分承接 |
| 2分 | 一致 | 0 | 可承接上游模块核心输出 |

**战略方向一致性判定**：
- 与品牌定位方向一致
- 与目标人群选择一致
- 与差异化策略方向一致

#### 5.4.7 输出状态枚举与判定原因枚举

**筛选状态（screening_status）枚举**：
- `passed`：通过筛选，进入优先级排序
- `watchlist`：进入观察池，待补充证据后复核
- `not_included`：不纳入正式机会清单
- `incomplete`：信息不完整，无法完成筛选判定

**判定原因（screening_reason）枚举**：
- `missing_required_fields`：缺少必填字段
- `insufficient_evidence`：证据不足
- `evidence_chain_incomplete`：证据链不完整
- `boundary_conflict`：边界冲突
- `cannot_form_downstream_input`：无法形成下游输入
- `duplicate_opportunity`：重复机会
- `insufficient_growth_potential`：增长潜力不足
- `insufficient_competition_feasibility`：竞争可切入性不足
- `insufficient_strategic_alignment`：战略承接性不足
- `meets_pass_threshold`：达到通过阈值
- `pending_additional_evidence`：待补充更多证据后复核

### 5.5 优先级排序规则

#### 5.5.1 四维权重

| 维度 | 权重 | 字段名 |
|------|------|--------|
| 需求真实度 | 20% | demand_validity_score |
| 增长潜力 | 30% | growth_potential_score |
| 竞争可切入性 | 25% | competition_feasibility_score |
| 战略承接性 | 25% | strategic_alignment_score |

#### 5.5.2 总分计算公式

```
total_score = (demand_validity_score × 0.20 + growth_potential_score × 0.30 + competition_feasibility_score × 0.25 + strategic_alignment_score × 0.25) × 100
```

**总分范围**：0-200分

#### 5.5.3 优先级等级划分

| 优先级等级 | 总分范围 | 判定条件 |
|------------|----------|----------|
| P1（立即推进） | total_score ≥ 150 | 所有维度 ≥ 1（无维度为0） |
| P2（计划推进） | 120 ≤ total_score < 150 | - |
| P3（观察储备） | 80 ≤ total_score < 120 | - |
| not_now（暂不推进） | total_score < 80 | - |

#### 5.5.4 一票降级硬性规则

- **任一维度 = 0**：最高只能到 P2，不能进入 P1
- **两个及以上维度 = 0**：最高只能到 P3

**硬性规则说明**：
- 任一维度为0表示该机会在对应维度存在根本性缺陷
- 即使总分达到150以上，只要任一维度为0，强制降级至P2
- 两个及以上维度为0表示多维度存在根本性缺陷，强制降级至P3

#### 5.5.5 阈值rationale

| 阈值 | 含义 | 计算依据 |
|------|------|----------|
| 150分 | 75%优秀线 | 满分200分 × 75% = 150分 |
| 120分 | 60%良好线 | 满分200分 × 60% = 120分 |
| 80分 | 40%潜力线 | 满分200分 × 40% = 80分 |

#### 5.5.6 同分处理规则

当多个机会项总分相同时，按以下顺序裁决：
1. 证据条数更多者优先
2. 证据条数相同 → 证据类型数更多者优先
3. 证据类型数相同 → `opportunity_id` 较小者优先

### 5.6 观察池管理规则

**复查周期（review_window）判定**：

| 周期 | 适用条件 | 所需补充证据类型 |
|------|----------|------------------|
| 30天（30d） | 数据补充型 | 缺失数据字段、规模估算、竞争评估 |
| 60天（60d） | 变化监控型 | 趋势变化、竞争动态、人群行为变化 |
| 90天（90d） | 长期验证型 | 需求持续性验证、市场成熟度变化 |

**复活条件**：
- 补充证据后满足筛选通过条件
- 市场环境发生重大变化导致原判定条件失效
- 上游模块输出更新带来新的支持证据

**观察池状态流转**：
```
进入观察池 → 补充证据 / 环境变化 → 重新筛选 → 通过 / 继续观察 / 淘汰
```

### 5.7 风险提示与假设管理规则

#### 5.7.1 风险提示表

| 风险类别（risk_category） | 定义 | 典型表现 |
|---------------------------|------|----------|
| 需求风险（demand_risk） | 需求真实性、持续性、强度存在不确定性 | 需求证据单一、需求季节性、需求可替代性高 |
| 竞争风险（competition_risk） | 竞争态势变化、竞争封锁、竞品反击存在不确定性 | 竞争封锁标记变化、竞品可能快速跟进、价格战风险 |
| 执行风险（execution_risk） | 内部执行能力、资源投入、时间窗口存在不确定性 | 新增前置条件多、执行复杂度高、时间窗口有限 |
| 战略风险（strategic_risk） | 与品牌战略一致性、战略优先级存在不确定性 | 与现有战略冲突、资源需求超出承受能力 |

#### 5.7.2 关联假设表

| 假设类型（assumption_type） | 定义 | 验证责任分配 |
|-----------------------------|------|--------------|
| 需求假设（demand_assumption） | 关于需求存在性、强度、持续性的假设 | 2.1 品牌定位 / 2.2 目标市场 |
| 竞争假设（competition_assumption） | 关于竞争态势、竞争封锁、竞品反应的假设 | 1.2 竞争分析（更新） |
| 能力假设（capability_assumption） | 关于内部能力、执行能力、资源能力的假设 | 6.x 运营模块 |
| 资源假设（resource_assumption） | 关于资源投入、资源获取、资源时效的假设 | 5.x 营销模块 / 6.x 运营模块 |

**假设验证责任分配原则**：
- 每个P1/P2级机会必须明确关联假设
- 假设验证责任分配给下游具体模块
- 假设未验证前，机会优先级不得提升
- 假设被证伪时，机会优先级必须重新评估

### 8.1 术语表

| 术语 | 定义 | 说明 |
|------|------|------|
| 机会项 | 进入机会分析前的字段化信号对象 | 不是正式机会结论，需经筛选后方可下游消费 |
| 机会池 | 所有候选机会项的集合 | 包含来自全部上游模块的机会信号 |
| 筛选标准 | 判定机会项是否进入正式机会清单的规则体系 | 包含证据链完整性、重复判定、四维评分 |
| 优先级排序 | 基于四维评分和一票降级规则的机会等级划分 | P1/P2/P3/not_now |
| 观察池 | 未进入正式机会主表、但满足观察池准入条件的机会项 | 需定期复查，补充证据后可复活 |
| 证据链 | 上游证据→问题缺口→动作方向的逻辑链条 | 必须满足字段一致性或可映射性 |
| 一票降级 | 任一维度为0时强制降低优先级的规则 | 硬性规则，不得突破 |
| 竞争封锁标记 | 描述竞争态势对机会切入的封锁程度 | open / partial_block / blocked |

| 下游映射 | 机会项可服务的下游模块标识 | 2.1 / 2.2 / 2.3 |

### 9.4 下游交接定义

1. **交给 2.1 品牌定位**：
   - `priority_opportunities`（优先级机会清单）
   - `opportunity_positioning_mapping`（机会-定位映射）
   - `differentiation_directions`（差异化方向）
   - `competitive_whitespace`（竞争空白点）

2. **交给 2.2 目标市场**：
   - `target_segment_opportunity_matrix`（目标人群-机会匹配矩阵）
   - `high_priority_segments`（高优先级人群）
   - `segment_opportunity_fit`（人群-机会匹配度）

3. **交给 2.3 差异化策略**：
   - `differentiation_opportunities`（差异化机会清单）
   - `competitive_gaps`（竞争缺口）
   - `value_proposition_candidates`（价值主张候选）

### 9.5 观察池联动规则

**选项A：独立维护 + ID关联**（本SOP采用）

- 1.5 观察池独立维护，不与 1.4 观察池合并
- 1.4 观察池中的对象进入 1.5 机会池时，标注 `derived_from_trend_watchlist = yes`
- 1.4 观察池对象在 1.5 中生成新的 `opportunity_id`
- 两池通过 `derived_from_trend_watchlist` 字段建立关联
- 1.5 观察池的复查周期独立计算，不受 1.4 观察池影响

### 9.6 阶段检查点（Checkpoint 1.5）

执行完毕后必须：
1. 输出"Phase 1.5 机会分析摘要"（正式机会数量、观察池数量、P1/P2/P3分布、Top 机会）
2. 展示正式机会主表、观察池、淘汰/暂存列表
3. 请求用户确认是否进入 2.x 品牌战略模块

**注意**：在用户确认前，**不得自动进入下一模块**。

  ]

```json

  "opportunity_watchlist": [
    {
      "opportunity_id": "OP010",

      "derived_from_trend_watchlist": "yes | no"
    }

```

**downstream_handoff**：
```json
{
  "downstream_handoff": {
    "for_2_1_positioning": {
      "priority_opportunities": ["string"],
      "opportunity_positioning_mapping": ["string"],
      "differentiation_directions": ["string"],
      "competitive_whitespace": ["string"]
    },
    "for_2_2_target_market": {
      "target_segment_opportunity_matrix": ["string"],
      "high_priority_segments": ["string"],
      "segment_opportunity_fit": ["string"]
    },
    "for_2_3_differentiation": {
      "differentiation_opportunities": ["string"],
      "competitive_gaps": ["string"],
      "value_proposition_candidates": ["string"]
    }
  }
}
```

**字段边界说明**：
- `opportunity_name`：面向下游展示的标准化机会名称，不得脱离证据新增判断
- `priority_level`：仅允许 `P1 / P2 / P3 / not_now`
- `total_score`：范围0-200，按5.5.2公式计算
- `downstream_mapping`：仅填写接收模块标识，不得在此字段写交付解释
- `evidence_summary`：必须按"【来源模块】+ 关键发现"格式记录证据摘要

**模块间数据存储**：
- 存储路径：`.sop/context/1.5-output.json`
- 存储格式：JSON
- 更新时机：本模块执行完毕且用户确认后
