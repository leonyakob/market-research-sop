## 前置声明

### "求真务实"原则（5条规则）

AI在执行本SOP时，**必须严格遵循以下规则**：

1. **所有趋势判断必须来源于可验证的数据、资料或上游模块输出，不得凭语言感觉补全趋势结论**。
2. **每一条正式趋势结论必须附带来源说明、归类依据、时间分层依据与下游映射说明**。
3. **禁止将热点、单次事件、孤立个案、未经验证的内部观察直接写为正式趋势**。
4. **当证据不足以支撑正式趋势归类时，必须明确标注为“长期观察方向”或“暂不纳入”，不得补全或猜测**。
5. **如不同来源存在明显冲突，必须按 `~/github/market-research-sop/交付文档/SOP-融合与冲突处理规则.md` 标注差异并处理，不得擅自平滑为单一结论**。

## 一、触发条件

### 1.1 何时执行本模块

本模块在以下场景触发执行：

- 已完成 1.1 / 1.2 / 1.3，需将行业、竞争、消费者变化信号转译为结构化趋势判断
- 在进入 1.5 机会分析前，需要先识别哪些变化可进入正式趋势主表，哪些仅进入长期观察池
- 在进入 2.x 品牌战略模块前，需要形成可复用、可下游消费的趋势接口

### 1.2 前置模块完成状态

| 前置模块 | 完成状态要求 | 验证方式 |

### 2.2 AI需确认（3-5个，含默认值机制）

| 参数 | AI询问方式 | 选项示例 | 默认值（仅用户明确要求时使用） |
|------|------------|----------|------------------------|
| 趋势输出粒度 | “本轮需要输出标准版还是扩展版趋势卡？” | 标准版 / 扩展版 | 标准版 |
| 时间窗口 | “趋势信号扫描窗口按近12个月、24个月还是36个月？” | 12 / 24 / 36个月 | 24个月 |
| 下游优先级 | “本轮趋势输出优先服务 1.5 机会分析，还是同时服务 2.x 品牌战略？” | 仅1.5 / 1.5+2.x | 1.5+2.x |
| 内部资料参与程度 | “内部资料仅用于解释，还是允许参与候选对象触发？” | 仅解释 / 参与触发 | 参与触发 |

**执行规则（强制交互）**：

1. **必须暂停**：AI 输出参数确认问题后，**必须立即停止生成**，不得继续执行搜索、分析或字段填充。
2. **等待回复**：等待用户明确回复（选择选项、输入自定义值或回复"使用默认值"）。
3. **禁止自作主张**：严禁在未获用户确认前使用默认值、补全参数或替用户推断任务口径。
4. **默认值触发**：仅当用户明确回复"使用默认值"或"跳过"时，方可使用默认值。
5. **报告中标注确认情况**：如“本研究参数已确认：时间窗口=近24个月；下游优先级=1.5+2.x”。

### 3.1 数据调用指令

**配置引用**：`.sop/config/data-acquisition.yaml`

**执行逻辑**：

```
读取 1.1 / 1.2 / 1.3 上游输出 → 生成自然语言查询 → 调用主数据源
  → 若主数据源失败 → 调用 fallback
  → 返回数据 → 提取趋势信号 → 字段化标注 → 进入有效性判断
```

**数据调用指令模板**（AI按此格式生成查询）：

| **消费者新兴行为** | `获取{品类}目标人群在{地理范围}近{N}个月出现的新购买行为、使用场景变化、偏好转移，包含具体行为、出现时间、涉及人群` | 行为信号列表 + 涉及人群 + 证据 |
| **文化与内容信号** | `获取{品类}相关消费文化、审美表达、话题偏好、内容传播方式在{地理范围}近{N}个月的变化，包含对应人群和场景` | 文化信号列表 + 内容表达变化 + 人群映射 |

- `{品类}` → 用户输入的目标品类

**当前主数据源**：Exa（`primary: exa`）  
**降级数据源**：Tavily（`fallback: tavily`）

**返回格式（当前）**：

```json
{
  "results": [
    {
      "title": "页面标题",
      "url": "https://...",
      "publishedDate": "2024-01-15",
      "text": "网页完整文本内容...",
      "highlights": ["AI提取的关键片段1", "关键片段2"],
```

**AI处理方式**：
1. 从返回的网页文本 / highlights / summary 中提取时间、对象、变化方向、驱动因素、受影响下游字段。
2. 将提取结果转写为候选对象，不直接写正式趋势结论。
3. 与 1.1 / 1.2 / 1.3 上游字段交叉验证。
4. 对每条候选对象补齐来源类别、时间覆盖、重复出现性、下游映射字段。

**配置文件引用**：`.sop/config/internal-files.yaml`

**文件读取规则**：

|----------|----------|----------|---------|

**内部文件标注规范**：

```
【内部-大类-子类_{标识}_{日期}.{格式}】
- 示例：【内部-用户观察-社媒话题变化_养生茶饮_202603.md】

```

**融合规则**：内部数据与外部数据并行比对，差异 <5% 标注“已验证”，差异 >10% 触发冲突预警。详见 `~/github/market-research-sop/交付文档/SOP-融合与冲突处理规则.md`。

### 3.5 AI自主补充查询

**补充触发条件表**：

| 条件 | 判断方法 | 阈值 | 补充查询指令 |
|------|----------|------|-------------|
| 上游字段缺口 | 1.1 / 1.2 / 1.3 任一关键字段缺失 | ≥1个关键字段缺失 | 补充查询：缺失字段对应维度 + 当前品类 |
| 时间覆盖不足 | 候选对象仅覆盖单一时间点 | 仅1个时间点 | 补充查询：同对象近{N}个月连续变化 |
| 来源类别不足 | 候选对象仅有1类来源 | 仅1类来源 | 补充查询：第二来源类别验证 |
| 方向解释不足 | 候选对象缺少驱动因素字段 | driver_label 为空 | 补充查询：对象名 + 变化原因 |
| 下游映射不足 | 候选对象无法映射到1.5或2.x | downstream_mapping 为空 | 补充查询：对象名 + 品牌战略/机会影响 |

**数据调用失败处理流程**：

```
主数据源调用失败 → 自动降级fallback（Tavily）
  → 同义词扩展查询（如“趋势”→“变化信号”→“行为变化”）
  → 扩大查询范围（如具体场景→所属消费场景大类）
  → 若仍无结果 → 标注“信息不足”并说明已尝试路径
  → 继续执行其他对象，不阻塞流程
```

## 四、输出格式

### 4.1 报告结构（6个部分）

**一、趋势信号来源与获取**
- 1.1 上游输入概览：1.1 / 1.2 / 1.3 已读取字段、可用性状态、缺口说明
- 1.2 外部补充数据概览：补充查询维度、来源类别、时效性说明
- 1.3 候选对象生成说明：候选对象数量、来源分布、对象类型分布

**二、趋势识别与聚类**
- 2.1 候选对象清单：对象名称、对象定义、target_object、direction_label、driver_label、source_categories
- 2.2 聚类结果：按 target_object + direction_label + downstream_mapping 分组后的聚类结果
- 2.3 聚类说明：合并原因、未合并原因、字段缺口说明

**三、伪趋势过滤 / 有效性判断**
- 3.1 有效性判断结果表：每个候选对象的通过 / 进入长期观察方向 / 暂不纳入结果
- 3.2 未通过对象清单：未通过原因、缺失字段、影响说明
- 3.3 长期观察方向清单：观察对象、观察原因、后续需补证据项

**四、时间分层**
- 4.1 短期趋势：已进入正式趋势主表的短期趋势对象
- 4.2 中期趋势：已进入正式趋势主表的中期趋势对象
- 4.3 长期观察方向：仅进入观察池，不进入正式趋势主表
- 4.4 暂不纳入：保留对象与排除原因

**五、趋势解释与影响分析**
- 5.1 趋势驱动解释：每条正式趋势对应的 driver_label 与证据来源
- 5.2 影响对象分析：受影响人群、场景、价格带、渠道、内容表达、品牌战略变量
- 5.3 风险与不确定性：证据缺口、冲突来源、需持续监测字段

**六、趋势输出与下游接口**
- 6.1 正式趋势主表：标准化趋势卡输出
- 6.2 长期观察池：观察对象与监测条件
- 6.3 淘汰/暂存列表：不进入正式输出的对象与原因
- 6.4 下游交接：1.5 机会分析、2.x 品牌战略可直接消费字段

### 4.2 数据集格式（4张表）

**表1：趋势候选对象表**

| object_id | object_name | signal_type | target_object | direction_label | driver_label | source_categories | upstream_source |
|-----------|-------------|-------------|---------------|-----------------|-------------|-------------------|----------------|
| T001 | 示例对象 | behavior / cultural / industry / channel / macro | 对象名 | increase / decrease / shift / split | 驱动标签 | external_report + internal_user_observation | 1.3.emerging_behaviors |

**表2：有效性判断结果表**

| object_id | repeat_count | source_category_count | external_support | downstream_mapping_count | validity_status | missing_fields |
|-----------|--------------|----------------------|------------------|--------------------------|-----------------|----------------|
| T001 | 3 | 2 | yes | 2 | validated | - |

**表2字段补充规则**：
- `repeat_count`：按同一 `target_object + direction_label` 在不同时间点出现的次数计算；仅统计不同月份的时间点，重复转载不重复计数
- `source_category_count`：按 `source_categories` 中 distinct 值数量计算，重复标签只计 1 次
- `external_support`：当 `source_categories` 包含 `external_report / external_news / external_platform_data` 任一值时填写 `yes`，否则填写 `no`
- `downstream_mapping_count`：按 `downstream_mapping` 中 distinct 模块标识数量计算
- `validity_status`：仅允许 `validated / watchlist / excluded`，不得在本表写入 `short_term / mid_term`
- `missing_fields`：仅填写缺失字段名数组或 `-`，不得写解释性长句

**表3：正式趋势主表**

| trend_id | trend_name | horizon | target_object | direction_label | driver_label | affected_people | affected_scenarios | affected_channel | affected_content_expression | downstream_mapping | evidence_summary |
|----------|------------|---------|---------------|-----------------|--------------|-----------------|-------------------|------------------|-----------------------------|-------------------|-----------------|
| TR001 | 示例趋势 | short_term | 对象名 | increase | ["健康焦虑上升", "渠道教育增强"] | 人群A | 场景A | 渠道A | 内容表达A | 1.5 / 2.1 | 【external_report】近12个月该对象在2个时间点重复出现，且最近6个月仍有外部支持 |

**表3字段补充规则**：
- `trend_name` 必须遵循 5.5 的生成规则；同一 `target_object + direction_label` 组合不得生成多个标准命名
- `horizon` 仅允许 `short_term / mid_term`
- `driver_label` 必须保留为驱动因素标签数组，不得并入 `trend_name` 或 `evidence_summary`
- `downstream_mapping` 仅允许填写模块标识，不得写解释句
- `evidence_summary` 必须按“【来源类别】+ 关键发现 + 时间覆盖”格式填写，不得写空泛总结句

**表4：长期观察池与淘汰/暂存表**

| object_id | list_type | object_name | observation_reason / exclusion_reason | required_next_evidence | review_window |
|-----------|-----------|-------------|--------------------------------------|------------------------|--------------|
| T010 | watchlist | 示例观察对象 | 缺少第二来源类别 | 补充第二来源类别验证，验证办公室代餐场景扩张，至少新增1类外部来源 | 90d |

**表4字段补充规则**：
- `list_type` 仅允许 `watchlist / excluded / pending`
- `required_next_evidence` 必须遵循 5.8 的标准模板
- `review_window` 仅允许 `30d / 60d / 90d`

### 4.3 数据来源标注规范

**数据融合标注**（v3.0沿用）：

| 标注格式 | 含义 | 示例 |
|---------|------|------|
| `【外部-数据源名】` | 仅单一外部源 | `消费者“无糖替代正餐化”信号【外部-艾瑞】` |
| `【外部-数据源名 + 内部-文件名】（已验证）` | 多源一致，差异<5% | `功能茶夜间饮用场景增加【外部-益普索 + 内部-渠道快报】（已验证）` |
| `【外部-数据源名 vs 内部-文件名】（差异X%，见冲突分析）` | 存在冲突，差异>10% | `该场景增速判断存在差异【外部-欧睿 vs 内部-销售月报】（差异15%，见冲突分析）` |
| `【信息不足】` | 搜索、读取或交叉验证后仍无法形成有效证据 | `某细分行为仅单一来源记录【信息不足】` |

**字段填写规则**：
- `object_name`：仅填写名词性对象名称，不得写结论句、比较句或带方向的判断句
- `object_definition`：仅用于说明对象边界，必须写明“包含什么 / 不包含什么”，不得重复填写趋势结论
- `target_object`：仅填写被分析的核心变化对象，不得混入受影响人群、渠道或策略建议
- `direction_label`：仅回答“对象朝哪个方向变化”，仅允许使用 `increase / decrease / shift / split / stabilize`
- `driver_label`：仅填写变化驱动因素标签，不得写方向判断、结果判断或主观评价
- `downstream_mapping`：仅填写既定接收模块或字段，不得写建议句、动作句或泛化结论

## 五、分析规则与决策标准

### 5.1 候选对象生成规则（强制）

每条候选对象必须先生成字段化对象，再进入后续判断。最小字段如下：

| 字段 | 类型 | 填写规则 |
|------|------|----------|
| `object_id` | string | 唯一编号，格式 `T001` 起 |
| `object_name` | string | 仅填写对象名称，不得包含趋势判断、比较关系、时间判断或结论性形容词 |
| `object_definition` | string | 仅用于说明对象边界，必须写明“对象包含什么 / 不包含什么”，不得改写为趋势结论 |
| `signal_type` | enum | `macro / industry / behavior / cultural / channel / competitive` |
| `target_object` | string | 仅填写被分析的核心变化对象；不得混入受影响人群、渠道标签、策略建议或结论句 |
| `direction_label` | enum | `increase / decrease / shift / split / stabilize` |
| `driver_label` | string[] | 仅填写驱动因素标签；不得写方向、结果、主观评价或泛化口号 |
| `downstream_mapping` | string[] | 仅填写固定模块标识，如 `1.5 / 2.1 / 2.3 / 5.2 / 5.3`；不得写解释句 |
| `source_categories` | string[] | 仅允许使用固定来源类别枚举：`upstream_1_1 / upstream_1_2 / upstream_1_3 / external_report / external_news / external_platform_data / internal_market_data / internal_competitor_intel / internal_user_observation / internal_strategy_note`；不得自造同义标签 |
| `source_evidence` | object[] | 每条证据必须包含来源、时间、摘要、字段对应关系 |
| `upstream_source` | string[] | 标注来自 1.1 / 1.2 / 1.3 的哪个字段 |

### 5.2 聚类规则（强制）

候选对象仅在以下字段一致时允许进入同一聚类：
1. `target_object` 完全一致
2. `direction_label` 完全一致
3. `downstream_mapping` 至少有 1 个相同值

**聚类后处理规则**：
- `driver_label` 可合并为多值数组
- `source_categories` 取并集
- `object_definition` 保留边界更清晰、排除条件更明确的版本
- 任一关键字段缺失时，不得强行聚类，保留为独立对象
- 不得按“语义相近”“同方向”“同类标签”进行主观合并

### 5.3 伪趋势过滤 / 有效性判断（强制前置）

候选对象进入时间分层前，必须先完成七维度有效性判断：

| 维度 | 判断方法 | 通过标准 |
|------|----------|----------|
| 重复出现性 | 同一 `target_object + direction_label` 是否在不同月份重复出现 | `repeat_count >= 2` |
| 来源独立性 | 是否至少来自2类不同来源类别（按 distinct 计数） | `source_category_count >= 2` |
| 外部支持 | `source_categories` 是否包含外部来源类别 | `external_support = yes` |
| 对象稳定性 | `object_definition` 是否可稳定复述为同一对象 | 对象边界无冲突 |
| 方向可支持性 | `direction_label` 是否有至少1条证据直接支持 | 方向标签可被证据复述 |
| 下游可映射性 | 是否能映射到至少1个下游模块字段 | `downstream_mapping_count >= 1` |
| 影响可描述性 | 是否能说明影响哪个人群/场景/渠道/内容表达之一 | 至少命中1类影响对象 |

**判定结果规则**：
- 7项全部通过 → 进入正式时间分层
- 缺少“重复出现性”或“来源独立性”任一项，但其他项通过 → 进入“长期观察方向”候选池
- 缺少“外部支持”或“方向可支持性” → 不得进入正式趋势或长期观察方向，直接“暂不纳入”
- 缺少“下游可映射性” → 暂不纳入正式趋势，保留在淘汰/暂存列表

### 5.4 时间分层规则（强制）

仅对已通过有效性判断的对象执行时间分层。

| 层级 | 进入标准 | 输出位置 |
|------|----------|----------|
| 短期趋势 | 近12个月内至少有2个时间点出现，且最近6个月仍有证据 | 正式趋势主表 |
| 中期趋势 | 覆盖12-36个月窗口，且最近12个月仍有证据 | 正式趋势主表 |
| 长期观察方向 | 未满足短期/中期，但满足观察池准入条件 | 长期观察池 |
| 暂不纳入 | 不满足观察池准入条件或缺少外部支持 | 淘汰/暂存列表 |

**长期观察方向准入条件**：
1. `external_support = yes`
2. `downstream_mapping_count >= 1`
3. `repeat_count = 1` 或 `source_category_count = 1`
4. 已记录 `required_next_evidence`

### 5.5 趋势命名规则（强制）

`trend_name` 必须在对象通过有效性判断并完成时间分层后生成，不得提前生成。

**生成公式**：
1. 以 `target_object` 作为命名主体
2. 以 `direction_label` 对应的固定中文方向词作为命名方向
3. 如需补充限定词，仅允许补充已被证据明确支持的场景、人群或渠道限定词
4. 不得加入未被字段证明的原因、价值判断、机会判断或策略暗示

**命名要求**：
- 必须写成可复述的标准化趋势名，不得写成口号、标题党、营销句或复合结论句
- 同一 `target_object + direction_label` 组合，在同一报告内只能生成 1 个标准命名
- `trend_name` 与 `object_name` 的区别：`object_name` 是候选对象名，`trend_name` 是正式趋势输出名；两者不得混用
- 若证据仅支持对象存在，不支持方向成立，则不得生成 `trend_name`

**固定方向映射**：
- `increase` → “{target_object}增加”
- `decrease` → “{target_object}减少”
- `shift` → “{target_object}迁移”
- `split` → “{target_object}分化”
- `stabilize` → “{target_object}稳定化”

### 5.6 趋势解释与影响分析规则

每条正式趋势必须补齐以下解释字段：
- `driver_label`
- `affected_people`
- `affected_scenarios`
- `affected_channel`
- `affected_content_expression`（如适用）

**执行边界**：
- `affected_price_band` 仅作为可选分析维度出现在正文解释中，不作为 `validated_trends` 强制字段
- `implication_for_1_5` 与 `implication_for_2_x` 统一通过 `downstream_handoff` 体现，不作为 `validated_trends` 强制字段

**填写规则**：
- 无证据支持时，填写 `unknown`，不得补全
- 若命中多个影响对象（≥2），允许数组输出，但不得混写为抽象总结句
- 至少 1 个影响字段必须为非 `unknown` 的具体对象，否则不得进入正式趋势主表


### 5.7 数据融合规则（强制）

**引用**：`~/github/market-research-sop/交付文档/SOP-融合与冲突处理规则.md`

**执行规则**：
1. 默认信任 1.1 / 1.2 / 1.3 正式输出，不做重建式重复分析。
2. 新补充证据与上游冲突时，检查口径、时间截点、样本范围。
3. 差异 <5%：可标注“已验证”并融合。
4. 差异 5-10%：保留上游值为主，新增差异说明。
5. 差异 >10%：双值并存，进入冲突标注，不得平滑为单值。

### 5.8 长期观察池字段规则（强制）

`required_next_evidence` 必须按“证据类型 + 验证对象 + 最低补强要求”的格式填写，不得写成空泛动作建议。

**标准模板**：
- `补充{证据类型}，验证{对象/方向}，至少新增{最低要求}`

**允许的证据类型**：
- 第二来源类别验证
- 连续时间点验证
- 外部支持验证
- 下游映射验证
- 影响对象验证
- 冲突消解验证

**最低要求写法**：
- `至少新增1类外部来源`
- `至少新增1个时间点`
- `至少补齐1个下游映射字段`
- `至少补齐1类影响对象字段`
- `至少补齐冲突差异说明`

**示例**：
- `补充第二来源类别验证，验证办公室代餐场景扩张，至少新增1类外部来源`
- `补充连续时间点验证，验证夜间饮用行为增加，至少新增1个时间点`

`review_window` 仅允许填写 `30d / 60d / 90d`，不得写为“近期复查”“持续关注”等自然语言。

### 5.9 降级模式（强制）

触发条件：
- 外部数据调用失败，无法获得外部支持
- 任一上游模块缺失正式输出

执行方式：
1. 仅生成候选对象列表与缺口说明
2. 不生成正式趋势主表
3. 仅允许输出“长期观察方向（低置信度）”与“暂不纳入”
4. 报告必须标注“本轮不用于高风险下游决策”

### 5.10 前置缺失处理规则

1. **1.1 缺失**：停止正式执行，不得输出正式趋势主表。
2. **1.2 缺失**：停止正式执行；仅在用户明确要求继续时进入有限探索模式，且不得生成正式下游交接。
3. **1.3 缺失**：停止正式执行；不得生成消费者与文化趋势相关正式结论。
4. **任一上游字段为空数组**：保留字段结构，标注 unavailable，不得补造内容。

### 9.5 阶段检查点（Checkpoint 1.4）

执行完毕后必须：
1. 输出“Phase 1.4 趋势分析摘要”（正式趋势数量、长期观察方向数量、Top 趋势）

### 9.6 输出数据格式规范（JSON Schema 参考）

**validated_trends**：
```json
{
  "validated_trends": [
    {
      "trend_id": "TR001",
      "trend_name": "string",
      "horizon": "short_term | mid_term",
      "target_object": "string",
      "direction_label": "increase | decrease | shift | split | stabilize",
      "driver_label": ["string"],
      "affected_people": ["string"],
      "affected_scenarios": ["string"],
      "affected_channel": ["string"],
      "affected_content_expression": ["string"],
      "downstream_mapping": ["1.5", "2.1"],
      "evidence_summary": ["string"]
    }
  ]
}
```

**字段边界说明**：
- `trend_name`：面向下游展示的标准化趋势名称，可由 `target_object + direction_label` 组合生成，但不得脱离证据新增判断，且必须遵循 5.5 的生成规则
- `horizon`：仅允许 `short_term / mid_term`；长期观察对象不得写入 `validated_trends`
- `target_object`：仅保留核心变化对象，不得混入影响对象或策略建议
- `driver_label`：仅保留驱动因素标签数组，不得替代 `trend_name`、`evidence_summary` 或 `downstream_handoff`
- `affected_people / affected_scenarios / affected_channel / affected_content_expression`：仅描述趋势影响范围；至少 1 个字段必须为非 `unknown` 的具体对象，不得替代 `target_object`
- `downstream_mapping`：仅填写接收模块标识，不得在此字段写交付解释；交付解释统一进入 `downstream_handoff`
- `evidence_summary`：仅按“【来源类别】+ 关键发现 + 时间覆盖”格式记录证据摘要

**long_term_watchlist**：
```json
{
  "long_term_watchlist": [
    {
      "object_id": "T010",
      "object_name": "string",
      "observation_reason": "string",
      "required_next_evidence": ["string"],
      "review_window": "30d | 60d | 90d"
    }
  ]
}
```

**字段边界说明**：
- `object_name`：沿用候选对象名称，不改写为趋势结论
- `observation_reason`：只写进入观察池的原因，不写正式趋势判断
- `required_next_evidence`：必须写明下一轮需要补充的证据类型或字段，不得写空泛描述
- `review_window`：仅填写复查周期，不表达判断结果

**模块间数据存储**：
- 存储路径：`.sop/context/1.4-output.json`
- 存储格式：JSON
- 更新时机：本模块执行完毕且用户确认后
